🎯 演算法核心優化

📏 尺度與長寬比幾何正規化

利用已知的行列數算出每個網格的物理邊界，並對去背碎片計算其最小外接矩形 cv2.minAreaRect。以長短邊比對長寬比，偏差過大直接跳過；再透過長邊比值計算精確縮放比例 scale_factor。將碎片縮放至與網格 1:1 等大後進行匹配。

⚡ 運算量降低 90%

利用外接矩形得到的碎片擺放角 $\theta$，僅需比對 $\theta$ 以及加上 90 度倍數的 4 個直角旋轉方向，不需進行盲目旋轉；同時 1:1 的縮放使得尺度搜尋降為 1 次。將原先 $15 \times 5 \times 8 = 600$ 次的匹配，縮減至 12 次以內，效率獲得飛躍性提升。

📐 仿射變換代替單應性

手機拍單片時透視畸變不嚴重，而在匹配點較少（4~8個）時使用 Homography 極易產生扭曲畸變框。改用 6 自由度的仿射變換 cv2.estimateAffine2D 作為備用，能有效約束平移、旋轉與縮放，防止投影框被拉伸變形。

🛡️ 形態學去凸耳與遮罩匹配

拼圖邊緣的凹凸卡榫會使外接矩形大 15%~25%，進而導致縮放本體被過度縮小。使用形態學開運算 (MORPH_OPEN) 削去凸耳，量測拼圖主體的純淨尺寸。並在比對時進行帶遮罩匹配，完全屏蔽去背黑色背景噪聲。

🌐 系統與 Web 架構規劃

1. 大圖特徵快取機制 (Feature Caching)

大圖通常高於 2560px，每次 API 請求都重新對大圖進行 SIFT 特徵計算在伺服器端是不切實際的。我們設計了二階段 API：

  • 專案建立 API：使用者上傳完成大圖，後端預先計算大圖特徵點與色彩矩，存入快取 (Redis 或記憶體中的 .npz 檔)。
  • 碎片定位 API：上傳碎片時直接從快取載入大圖特徵，定位回應時間可從數秒壓縮至 100~300ms。

2. 數據與圖片分離（前端 Canvas 繪製）

後端定位 API 不再傳送畫好框的大圖，而是回傳輕量級的 JSON 幾何座標與網格 metadata。前端網頁載入一次大圖後，透過 HTML5 Canvas / SVG Overlay 動態在網頁上疊加定位框與多候選切換，大大減少頻寬消耗，提升流暢性。

💡 前端 UI/UX 優化方案： 當定位完成時，UI 可以顯示一個 3D 拼圖旋轉提示（例如：「請順時針旋轉 90 度」），並在 Canvas 定位框上附帶向外擴散的半透明雷達定位波，提供極具科技感的 premium 互動體驗。

🛠️ 實作修改內容

我們已經成功在專案的 source/features/localization/locator.py 中完成了上述演算法的重構實作。

1. 新增 Helper 函數

在 locator 模組中，新增了兩個核心 Helper 函數來解決幾何跳變與卡榫問題：

  • _standardize_rotated_rect(rect)：強制將外接矩形的 width 鎖定為長邊，height 鎖定為短邊，並精確計算標準旋轉角。
  • _get_puzzle_body_rect(mask)：在遮罩上以橢圓形結構元素進行開運算開路，剔除卡榫凸耳，取得拼圖主體的精確尺寸。

2. 重構 4 向帶遮罩模板匹配

在 SIFT 失敗退路時，利用長寬比快速過濾，接著以長邊計算 scale 進行 1:1 圖像 resize。將網格搜尋區向外擴大 1.25 倍防止旋轉越界，並僅旋轉 4 個直角方向進行帶遮罩匹配 cv2.matchTemplate(..., mask=rotated_mask)。
